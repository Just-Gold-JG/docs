# Webhooks

JustGold sends webhook calls to your configured endpoint when important events happen in your integration.

Your endpoint must verify the webhook signature before processing the event.

## Supported events

- `transaction.created`
- `transaction.status_changed`
- `customer.created`

## Webhook request

JustGold sends webhook events as `POST` requests with a JSON body.

```http
POST <your_webhook_url>
Content-Type: application/json
X-Timestamp: 1767225600
X-Signature: <webhook_signature>
```

## Headers

Verify these headers on every webhook request:

| Header | Required | Description |
| --- | --- | --- |
| `X-Timestamp` | Yes | Unix timestamp in seconds generated by JustGold. |
| `X-Signature` | Yes | HMAC-SHA256 signature generated by JustGold. |

## Payload format

Each webhook payload includes the event type and a flexible payload object.

```ts
export interface WebhookPayload {
  eventType: WebhookEventType;
  payload: {
    [key: string]: unknown;
  }
}
```

### `transaction.created`

Sent when a new transaction is created, regardless of its initial status.

| Field | Type | Description |
| --- | --- | --- |
| `id` | string | Transaction identifier. |
| `type` | string | Transaction type. One of `Buy`, `Sell`, or `Delivery`. |
| `metal` | string | `Gold` or `Silver`, when applicable. |
| `status` | string | Initial transaction status. Typically `Pending`. |
| `quantity` | string | Metal quantity in grams. |
| `customerId` | string | Customer identifier. |
| `organizationIds` | array | Organization hierarchy associated with the transaction. |
| `currency` | string | Transaction currency. |
| `quotedPrice` | string | Quoted price per gram. |
| `vat` | string | VAT amount applied to the transaction. |
| `platformFee` | string | Platform fee applied to the transaction. |
| `total` | string | Transaction total before fees. |
| `grandTotal` | string | Final total including all fees. |
| `notes` | string | Notes stored with the transaction, when set. |
| `paymentReference` | string | Partner payment reference, when set. |
| `paymentMethod` | string | Payment method, when set. |
| `settlement` | object | Settlement breakdown, when available. See below. |
| `createdAt` | string | Transaction creation timestamp (UTC). |
| `updatedAt` | string | Transaction last update timestamp (UTC). |

### `transaction.status_changed`

Sent when a transaction status changes (e.g. `Pending` → `Completed`).

| Field | Type | Description |
| --- | --- | --- |
| `transactionId` | string | Unique transaction identifier. |
| `type` | string | Transaction type: `Buy`, `Sell`, or `Delivery`. |
| `status` | string | New status: `Pending`, `Completed`, `Failed`, or `Cancelled`. |
| `customerIdentifier` | string | Partner-scoped customer identifier. |
| `metal` | string | `Gold` or `Silver`. |
| `quantity` | string | Metal quantity in grams. |
| `currency` | string | Transaction currency (e.g. `AED`). |
| `price` | string | Actual price per gram at the time of transaction. |
| `total` | string | Pre-fee amount (quantity × price). |
| `platformFee` | string | Platform fee applied. Omitted if zero. |
| `vat` | string | VAT amount applied. Omitted if zero. |
| `grandTotal` | string | Final amount after fees and VAT. |
| `organizationId` | string | Organization that processed the transaction. |
| `createdAt` | string | ISO 8601 timestamp (UTC). |

## Signature verification

Webhook signatures are generated with HMAC-SHA256 using your `client_secret`.

Build the string to verify with these exact values:

```text
{TIMESTAMP}
{RAW_REQUEST_BODY}
```

Rules:

- `TIMESTAMP` must match the value received in the `X-Timestamp` header.
- `RAW_REQUEST_BODY` must be the exact raw request body received by your endpoint.
- Do not parse, reformat, or stringify the JSON before verifying the signature.
- Process the webhook only if the signature is valid.

## Verification algorithm

1. Read the raw request body.
2. Build the string to verify.
3. Compute `HMAC-SHA256(string_to_verify, client_secret)`.
4. Encode the digest as lowercase hexadecimal.
5. Compare it with the value received in `X-Signature`.
6. Process the event only if the signatures match.

## Node.js example

```js
import crypto from "node:crypto";

function verifyWebhookSignature({
  timestamp,
  body,
  receivedSignature,
  clientSecret,
}) {
  const rawBody = body ?? "";
  const stringToVerify = [
    String(timestamp),
    rawBody,
  ].join("\n");

  const expectedSignature = crypto
    .createHmac("sha256", clientSecret)
    .update(stringToVerify, "utf8")
    .digest("hex");

  if (expectedSignature.length !== receivedSignature.length) {
    return false;
  }

  return crypto.timingSafeEqual(
    Buffer.from(expectedSignature, "utf8"),
    Buffer.from(receivedSignature, "utf8"),
  );
}
```

## Verification checklist

- Reject webhook requests with missing `X-Timestamp` or `X-Signature` headers.
- Verify the signature before parsing the payload into business logic.
- Use the exact raw request body received by your server.
- Use a constant-time comparison for signatures.
- Reject timestamps outside your allowed clock-skew window.
- Log verification failures without logging the `client_secret`.

## Handling webhooks

- Return a `2xx` response only after your endpoint accepts the event.
- Use `eventType` to determine how to process the `payload`.
- Do not trust the payload until signature verification succeeds.
- Keep webhook processing fast. For longer work, store the event and process it asynchronously.
